# 第四章：工具系统

工具不是函数，是声明式契约。这个认知转变，是构建可靠 Agent 的关键一步。

---

## 设计动机：从函数到契约的认知转变

大多数开发者第一次做工具系统时，思路是"把函数包装成工具"。这个思路能跑起来，但很快会遇到一面墙：

框架需要知道，但函数无法表达的信息：
- 这个工具可以和其他工具并发执行吗？（并发安全性）
- 执行前需要用户确认吗？（权限模型）
- 子 Agent 可以调用这个工具吗？（工具可见性边界）
- 计划模式下应该怎么处理？（执行语义）
- 结果太大时应该怎么处理？（输出上限）

这些信息不是执行逻辑的一部分，是工具的元数据。把元数据和执行逻辑放在一起，就得到了"声明式工具契约"：工具不只告诉框架"我能做什么"，还告诉框架"框架应该怎么用我"。

---

## 业内工具系统设计对比

| 框架 | 工具定义方式 | 并发控制 | 权限模型 | 动态工具集 |
| --- | --- | --- | --- | --- |
| LangChain | Python 函数 + 装饰器 | 无内置机制 | 无内置机制 | 支持运行时注册 |
| OpenAI Function Calling | JSON Schema 声明 | 无内置机制 | 无内置机制 | 每次调用传入 |
| Claude Tool Use | JSON Schema + description | 无内置机制 | 无内置机制 | 每次调用传入 |
| AutoGen | Python 函数 + register_function | 无内置机制 | 无 | 运行时注册 |
| Hermes | Python 装饰器 + registry | 无`isConcurrencySafe` | 无内置机制 | 运行时 AST 扫描发现 |
| Alice | TypeScript 对象 + 元数据字段 | `isConcurrencySafe`（含动态判断） | `requiresPermission`字段 + 五层模式 | 动态组装（按 Skill/角色/设置过滤） |

**Alice 的关键设计：`isConcurrencySafe` 可以是函数**
大多数工具的并发安全性是固定的（搜索永远安全，写文件永远不安全）。但有些工具的安全性取决于输入：比如"写入到不同文件"可以并发，"写入到同一文件"不能。允许`isConcurrencySafe`是一个根据输入动态计算的函数，让框架能做出更精细的编排决策。

---

## 为什么不能只是函数

当工具数量少、场景简单时，纯函数没问题。但当你开始面对这些问题，就会发现纯函数不够用：
- 这个工具可以和其他工具并发执行吗？
- 执行前需要用户确认吗？
- 结果太大时应该怎么处理？
- 子 Agent 可以调用这个工具吗？
- 计划模式下（只规划不执行）应该怎么处理？
- 这个工具是只读的还是会修改状态？

工具是声明式契约：它不只是执行代码，还包含关于自身行为的元数据声明，让框架可以做出正确的编排决策。

---

## 工具声明的关键字段

一个完整的工具声明应该包含：
- **name**：工具唯一名称（LLM 用它识别工具）
- **description**：功能描述（直接影响 LLM 的调用决策！）
- **inputSchema**：输入格式（用于类型验证和 JSON Schema 生成）
- **isReadOnly**：是否只读（不修改任何状态）
- **isDestructive**：是否有不可逆影响（删除文件、发送消息等）
- **isConcurrencySafe**：是否可以与其他工具并行执行
- **requiresPermission**：执行前是否需要权限检查
- **maxResultSizeChars**：输出大小上限（超出则写文件返回路径）
- **execute(input, ctx)**：实际执行函数

---

## description 是最重要的字段

工具描述的质量直接决定 LLM 是否会在正确的时机调用它。

**正例（包含触发时机、使用场景、不适用场景）：**
```
description: |
  将重要信息持久化到项目记忆文件。

  应该使用的场景：
  - 用户明确要求"记住"某件事
  - 发现了项目的特殊约定或规则
  - 完成了重要任务，需要记录结果供未来参考

  不应该使用的场景：
  - 临时性信息（这次对话用完就不需要了）
  - 可以从代码文件直接查到的信息
  - 会频繁变化的动态数据
```

---

## 并发调度：isConcurrencySafe 的价值

`isConcurrencySafe` 提供了中间路径：
```
调度逻辑：
把工具调用列表分成批次：
  - 连续的"并发安全"工具合为一批，并行执行
  - "不安全"的工具单独一批，串行执行
```

---

## 执行上下文注入（ToolContext）

工具代码不依赖任何全局变量，所有需要的东西通过 `ctx` 注入。这让工具可以在任何环境里运行，也让工具的依赖关系清晰可见。

---

## 大结果落盘

某些工具可能返回几十 MB 的内容。处理方案：
1. 检查结果大小是否超过 `maxResultSizeChars`
2. 超过则写入临时文件
3. 返回："结果已写入 /tmp/result_xxx.txt，使用文件读取工具查看"

---

## 子 Agent 的工具限制

子 Agent 不应该拥有与父 Agent 相同的完整工具集。需要禁止的工具包括：
- `agent`：禁止递归创建孙子 Agent
- `ask_user`：子 Agent 不能直接弹窗问用户
- 计划模式控制工具
- 渠道发送工具
- 记忆写入工具
