# 第六章：多 Agent 协作

一个 Agent 看世界是串行的，多个 Agent 可以并行感知。但更关键的问题是：信息如何流动。

---

## 设计动机：为什么不用一个更大的 Agent

多 Agent 的决策前提是：先判断任务是"信息积累型"还是"并发执行型"。
- **信息积累型**：单 Agent + 好的上下文管理通常更好。
- **并发执行型**：需要多 Agent 解决推理质量下降和串行执行限制。

---

## 三种协作模式

### 模式一：父子（Subagent）
父 Agent 通过 `agent` 工具创建子 Agent。
- **关键设计**：`isConcurrencySafe` 由 `isReadOnly` 参数决定。只读任务可并行，写任务需串行。

### 模式二：Coordinator（协调者）
Coordinator 负责任务分解和结果综合，Worker 负责执行。
- **原则**：优先研究再行动；读写分离；不委托理解（Coordinator 必须自己综合结果）。

### 模式三：Swarm（群体）
去中心化协作。多个对等 Peer 共享一个任务队列（SQLite 持久化），各自领取任务执行。
- **核心**：原子性领取任务；显式声明任务依赖关系。

---

## 子 Agent 的隔离机制

1. **进程隔离**：运行在独立 Worker Thread 或进程。
2. **上下文隔离**：不与父 Agent 共享 ContextManager。
3. **工具隔离**：使用受限工具集。
4. **降级机制**：Worker Thread 不可用时降级到主线程运行。

---

## 权限的向下传递

子 Agent 的权限不能高于父 Agent。创建时传递 `permissionMode`，子 Agent 只能继承或降级。

---

## 信息流：多 Agent 的核心挑战

**反模式**：通过共享全局状态传递信息（易产生竞态）。
**正确模式**：结构化文本返回，Coordinator 负责综合。每个 Agent 的职责是"完成任务，返回结果"。

---

## Alice 三模式的选择逻辑

- **父子模式**：任务边界清晰、子任务数量已知、父 Agent 有能力综合。
- **Coordinator 模式**：任务复杂，分解本身需要推理；有明确依赖顺序。
- **Swarm 模式**：任务边界动态产生；需跨会话持久化进度。
