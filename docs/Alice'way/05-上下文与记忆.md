# 第五章：上下文与记忆

"上下文焦虑"是 Agent 产品的核心难题，但业内的解法差距悬殊。

---

## 为什么这是个难题：设计动机

LLM 本质上是无状态的。上下文窗口就是它的"短期记忆"，一旦关闭，什么都不剩。

有两个相互独立的子问题：
1. **上下文管理**：解决"当前这次对话的信息放不下"的问题。工具：压缩、截断、摘要。
2. **记忆系统**：解决"上次对话的重要信息怎么带到这次对话"的问题。工具：持久化存储、检索、注入。

---

## 第一部分：上下文管理

### 四层压缩策略

按上下文使用率从低到高，依次触发更重量级的处理：

1. **层一：Snip**
   最轻量，成本最低：删除最早的几对工具调用轮次。保护任务说明（早期消息）不被删除。

2. **层二：MicroCompact**
   本地替换，零 API 成本：对重复出现的相同工具调用，保留最后一次结果，替换早期重复调用为一行简短说明。

3. **层三：Collapse**
   调用 LLM 做摘要，保留近期消息的完整性。摘要应该是结构化的（当前任务、已完成步骤、当前状态、下一步）。

4. **层四：AutoCompact**
   最重量级，用独立的 LLM 会话对全量历史做完整摘要。需处理递归陷阱（禁用工具、标记压缩状态）。

---

## 第二部分：记忆系统

### 五层记忆架构

不同类型的信息有不同的读写频率和生命周期：

| 层 | 名称 | 位置 | 生命周期 | 存什么 |
| --- | --- | --- | --- | --- |
| M1 | 在线上下文 | 内存 | 单次对话 | 当前工作状态、工具调用历史 |
| M2 | 项目记忆 | 文件系统 (ALICE.md) | 项目级 | 约定、规则、重要决策 |
| M3 | 向量记忆 | 向量数据库 | 全局 | 历史对话的语义化存储 |
| M4 | 结构化存储 | SQLite | 持久化 | 会话列表、消息历史、设置 |
| M5 | 用户画像 | 文件 (memory/*.md) | 长期 | 用户身份、偏好、风格 |

### M2：项目记忆的层级读取
项目记忆文件按目录层级组织（从全局到项目到子目录），越深层的规则优先级越高。

### M3：向量记忆的写入时机
对话进行中产生的消息不能立即写入向量库，需在整轮对话结束后统一 flush，防止信息自我强化循环。

### M5：用户画像的三个维度
- **identity.md**：用户是谁
- **workflow.md**：用户怎么工作
- **voice.md**：用户怎么沟通

---

## 上下文工程的本质

上下文工程的核心是**"什么时候装什么"**，而非单纯追求"装多少进去"。信息的价值随时间和场景而变化，没有一种存储方式对所有类型的信息都最优。
