# Phase 3: 基于五大设计哲学的重构与迭代计划

## 核心目标

参考 Alice 的五大设计哲学，对当前系统（特别是臃肿的 `runAgent.ts` 及缺乏追踪的调度逻辑）进行深度重构与功能迭代，使其达到生产级的高内聚、低耦合、高可观测性标准。

---

## 哲学一：接口长寿 (Interface Longevity)

### 1.1 事件流规范化 (Standardized Event Stream)
**功能描述**：当前 Agent 的输出主要依赖非结构化的 `sendState` (字符串) 和 `sendChunk` (文本流)。需要定义一套严格的、向后兼容的事件接口，如 `tool_start`, `tool_end`, `error`, `text_chunk`, `agent_finish`。UI 层只消费结构化事件。
**测试用例 (TC-1.1)**：
- **操作**：模拟触发一个需要 3 秒的工具调用。
- **断言**：前端依次收到类型为 `tool_start` 的事件，包含工具名和入参；随后收到 `tool_end` 事件，包含结果 Payload。前端 UI 渲染不受内部逻辑变更影响。

### 1.2 LLM 提供商适配层 (LLM Provider Adapters)
**功能描述**：从 `runAgent.ts` 中剥离 OpenAI 与 Anthropic 的特定请求组装与解析逻辑，抽象出统一的 `LLMAdapter` 接口。主循环只面向 `LLMAdapter` 编程，屏蔽模型差异。
**测试用例 (TC-1.2)**：
- **操作**：切换环境变量中的 `DEFAULT_LLM_PROVIDER` 从 `minimax` 到 `anthropic`。
- **断言**：Agent 依然能正确识别工具调用（由适配层将 Anthropic 的 `tool_use` 统一转换为内部标准的 `ToolCall` 对象），执行完毕后能正常返回给前端，整个过程不需要修改 `AgentLoop`。

## 哲学二：状态优先 (State First)

### 2.1 状态一致性保护与并发控制
**功能描述**：在多个工具或异步任务（如 Share Extension 触发的 Explore Agent）尝试修改长期记忆或会话状态时，引入锁机制或事务机制，防止状态冲突和写入丢失。
**测试用例 (TC-2.1)**：
- **操作**：模拟并发发起两个需要修改同一个 Session 历史记录和长期记忆的请求。
- **断言**：数据库中的 Message 序列保持正确先后顺序，未发生数据覆盖或丢失，并发写操作被正确串行化或合并。

### 2.2 结构化历史压缩与状态提取
**功能描述**：优化现有的 `compressHistoryIfNeeded`。不只是做简单的文本摘要，而是将关键事实提取到独立的 `WorkingMemoryState` 对象中（如用户偏好、当前未完成任务列表），并在系统提示词中结构化恢复。
**测试用例 (TC-2.2)**：
- **操作**：注入超过 Token 限制的冗长对话历史，其中包含一条“提醒我晚上买牛奶”的未完成任务。
- **断言**：触发压缩后，最新的 `messages` 数组被截断，但压缩后的结构化上下文明确包含了 "未完成任务: 晚上买牛奶" 字段，且 Agent 在后续回答中能准确接续该任务。

## 哲学三：结构隔离 (Structural Isolation)

### 3.1 `runAgent.ts` 拆库与职责分离
**功能描述**：将长达 200 多行的 `runAgent` 函数拆解。抽象出 `AgentLoop` 类负责多轮 ReAct 调度，`PromptBuilder` 负责上下文组装，`ToolHarness` 负责工具执行拦截与通知。
**测试用例 (TC-3.1)**：
- **操作**：对拆分后的核心类进行单独实例化。
- **断言**：能够脱离 WebSocket 环境，直接在纯 Node.js 测试脚本中调用 `AgentLoop.run` 并传入 Mock 的上下文，正确输出预期回答，验证核心逻辑与网络传输完全解耦。

### 3.2 区分内置服务与 AI 工具
**功能描述**：审查所有工具，确保没有任何“框架内部所需服务”被错误地暴露给 AI。例如内存快照、会话持久化等应由生命周期钩子（Hooks）或内部 Service 完成，严格向 AI 隐藏。
**测试用例 (TC-3.2)**：
- **操作**：打印并列出系统启动时 AI 可见的所有 `availableTools`。
- **断言**：列表中仅包含对用户意图有直接意义的操作（如 `AskClarification`, `SearchNotion`），不包含任何内部控制或框架生命周期相关的工具接口。

## 哲学四：可丢弃组件 (Disposable Components)

### 4.1 LLM 降级与兜底路由 (LLM Fallback Router)
**功能描述**：主模型（如 MiniMax-M2.7 或 Claude）调用失败、超时或触发频率限制时，系统应自动捕获并降级调用备用模型（如本地部署的小模型或成本更低的 API）。
**测试用例 (TC-4.1)**：
- **操作**：将主模型 API URL 拦截并强制返回 500 服务器错误或超时。
- **断言**：系统在指定时间内捕获异常，记录降级日志，并成功使用次选模型完成对话回复，用户端不出现服务崩溃。

### 4.2 非关键服务（如记忆检索）的优雅降级
**功能描述**：当基于向量的深度记忆搜索或数据库查询不可用时，系统降级为简单的关键字匹配或内存中近期的会话检索，保证“质量可以下降，但系统不能崩溃”。
**测试用例 (TC-4.2)**：
- **操作**：强制断开 Vector DB / Embedding 服务的连接，向 Agent 询问长期记忆中的问题。
- **断言**：Agent 不崩溃，控制台记录“降级到本地缓存匹配”，通过本地文本正则匹配寻找答案，并给予用户兜底回复。

## 哲学五：可观测性 (Observability)

### 5.1 结构化 Trace 与 Span 日志
**功能描述**：引入轻量级链路追踪。每次用户的 Query 生成一个唯一的 `trace_id`。从分析需求、记忆召回、LLM 思考到工具执行，产生多个结构化的 `span_id`，统一记录到日志文件，供可视化和回溯。
**测试用例 (TC-5.1)**：
- **操作**：发起一轮包含 2 次工具调用的复杂对话。
- **断言**：系统日志中能够查询到该 `trace_id` 对应的完整生命周期 JSON 树，明确记录了每个子阶段（Span）的耗时（ms）和输入输出 Payload。

### 5.2 强制思维过程可观测 (Chain of Thought Logging)
**功能描述**：在使用某些不自带内部思维过程输出的模型时，在系统提示词或工具 schema 中强制要求其输出 `reason` 字段，表明本次操作的目的和依据，避免推理黑盒化。
**测试用例 (TC-5.2)**：
- **操作**：要求 Agent 调用 Notion 工具保存某个信息。
- **断言**：Agent 发起的工具调用 payload 中，强制且清晰地包含了类似 `"reason": "用户明确要求将上述总结存入Notion"` 的字段，日志可审计该决策。

## 第三阶段 (Phase 3) 实施路线图与测试要求

| 优先级 | 里程碑 | 涵盖功能项 | 预计周期 | 前置依赖 |
|---|---|---|---|---|
| **P0** | **基础设施：可观测性建设** | 5.1, 5.2 | 1 个迭代 | 无 |
| **P0** | **核心重构：Agent拆解与解耦** | 1.2, 3.1, 3.2 | 2 个迭代 | 5.1（需依赖日志确保重构正确性） |
| **P1** | **契约升级：事件流与状态抽象** | 1.1, 2.2 | 1 个迭代 | 3.1 |
| **P1** | **系统韧性：降级与兜底** | 4.1, 4.2 | 1 个迭代 | 1.2 |
| **P2** | **高级状态：并发控制** | 2.1 | 1 个迭代 | 2.2 |

> **研发纪律**：在开发上述任一功能项前，开发人员必须先实现对应编号的 TC (Test Case)（自动化或明确的断言文档步骤）。只有在测试用例覆盖下进行的重构和迭代，才能保证严格遵循 Alice 的设计哲学。
