# 第一章：Agent 的大脑与思维（System Prompt 与 ReAct 循环）

本章我们深入探讨了 Agent 的底层技术原理，并从 AI 产品经理的视角拆解了这些技术如何映射为用户价值和产品体验。以下是关于大脑核心机制、记忆管理以及多 Agent 协同的深度 Q&A 记录。

## 1. System Prompt（系统提示词）与 Prompt Cache

### Q：静态区和动态区的拼接，是这个 Session 下的每一轮对话都会做吗？为什么？
**A：** 是的，是每一轮对话都会重新拼接发送。
因为大模型本质上是“无状态的（Stateless）”，它没有记忆。为了让它保持对人设和当前环境的感知，必须在每次发请求时，把“静态人设 + 动态上下文 + 之前的历史对话记录”全量发送过去。

### Q：提示词缓存（Prompt Cache）是怎么做到的？MiniMax 支持吗？我们的框架在用吗？
**A：** 
*   **产品视角的原理解释**：你可以把大模型想象成“现炒餐厅”，Prompt Cache 就是把几万字的《菜谱》（静态人设和工具库）提前贴在厨师的案板上。后续只要前缀匹配，大模型就会“瞬间读取”缓存，首字响应速度（TTFB）能提升 2-3 倍，且这部分缓存的 API 费用通常打一折。
*   **MiniMax 支持情况**：支持。MiniMax M2.7 等模型支持类似 Anthropic 的显式 Prompt Cache。
*   **框架现状**：在我们的 MVP 框架计划中（Epic 1.2.2），已经规划了在 `STATIC_PERSONA` 结尾打上 Cache 标记以复用长文本人设。

### Q：我的 API Key 是外接的服务，为什么会和其他人共享几万字工具和规则的缓存？
**A：** 这里的“共享”不是指和其他公司共享，而是指**你的产品下的所有真实用户，在共享你的 API Key 缓存**。因为所有用户发送给大模型的前半段（系统设定和工具库）是完全一致的，只要一个用户触发了缓存，几分钟内其他用户的请求也会命中这个缓存，极大地摊薄了 C 端高并发产品的整体成本。

### Q：如果工具说明书放在静态区缓存，首轮提问的响应速度是不是大概率比后续追问更长？
**A：** 如果**没有**缓存，首轮肯定最慢。但如果**用好了**缓存，首轮提问因为命中了云端缓存，读取速度是毫秒级的，反而极快。而后续的追问，由于聊天记录（History）不断变长（这部分是动态增长、无法完全前置缓存的），LLM 每次都要重新阅读变长的 History，导致越到后面的追问，处理时间反而可能越长。

## 2. 上下文窗口与记忆管理 (Working Memory)

### Q：如果每次对话都提交，上下文窗口岂不是很快过载？会不会遗忘？框架如何解决？
**A：** 是的，一定会过载，遗忘是物理限制。
为了解决这个问题，我们在架构中引入了**压缩与裁剪（Working Memory/Episodic Memory）**机制：
当对话进行到一定长度，系统会在后台唤醒一个小模型，对前面的冗长对话进行“压缩和提炼”（比如提取成一句话总结）。然后把原始的长聊天记录从上下文中删掉，只把这段浓缩的“工作记忆”拼接到“动态区”发给大模型。这样既占用了极小的 Token 空间，又维持了对全局任务状态的清晰认知。

## 3. Tools、MCP、Skills 的生态占位

### Q：Tools、MCP、Skills 有什么区别？它们是不是都会被拼接到每次提问中？
**A：** 它们是 Agent 能力的三个不同层级，如果太长会通过 Prompt Cache、按需检索 (Tool RAG) 或外包给 Subagent 来解决上下文过长的问题。
*   **MCP（生态）**：模型上下文协议，是“外包资源库 / 通用插排”。它不是具体工具，而是标准协议，让系统能瞬间自动接入三方软件（如 Notion）的一整套 Tools。
*   **Tools（下限）**：原子工具，是 Agent 的“手脚”。比如发邮件的代码函数。Tools 决定了 Agent “能做什么”，保证了任务执行的下限。
*   **Skills（上限）**：操作指南 / SOP。比如《如何写商业邮件》。它是用文字或示例代码写成的规范，当触发特定任务时作为上下文注入，教 Agent “如何把事情做得专业漂亮”，决定了任务交付的上限。

## 4. Agent Loop 与 Agentic 架构演进

### Q：Agent Loop 本质是 while 循环，是不是模型越好，Loop 的准确性越好、步骤规划越合理，产品感知上 Agent 就越聪明越快？
**A：** 完全正确！强大的模型（如 Claude 3.5 Sonnet）具备极强的规划能力（Planning），它能合并工具调用、减少试错的弯路。需要的 Loop 次数越少，体感反应就越快，这也正是“聪明”的体现。

### Q：为什么厂商（如 MiniMax）说模型的自主迭代 Harness 的能力很重要？这不应该是架构层写在 System Prompt 里的事吗？
**A：** 把规则全写在 System Prompt 里有“认知天花板”（模型听不懂太复杂的规矩，陷入死循环也不会自救）。顶级厂商正在把 Agent 能力**内化到模型权重**里。出厂自带 Agent 能力的模型，不仅能自己应对复杂的工具调用流，还能发现架构（Harness）的漏洞并自主优化代码。这意味着产品经理在 System Prompt 里需要写的“防御性废话”越来越少。

## 5. Subagent（子代办）与多 Agent 协同

### Q：什么时候调用 Subagent？它和主 Agent 的工具集、模型是一样的吗？
**A：** 
*   **调用时机**：当任务繁琐（如扫读 50 个网页）时调用。Subagent 负责隔离一次性的“垃圾上下文”，提取精华后销毁自己，保持主 Agent 大脑（Context）的清爽。
*   **工具集**：Subagent 可以有**独有**的工具集，实现权限和能力隔离。如果它发现工具不够用，通常会向上抛出异常，让主 Agent 接手或重新派发。
*   **模型选择**：为了降本增效，主 Agent 负责高情商沟通和统筹，使用最贵最聪明的模型；而 Subagent 负责脏活累活，可以使用便宜极快的小模型。

### Q：如果有多个 Subagent，主 Agent 和它们是并行还是串行？主 Agent 会自己做其他事吗？
**A：** 既有串行也有并行。
在最前沿的 Agentic Workflow 中，主 Agent 收到复杂任务后，会将其拆解并**并发（并行）**下发给多个不同的 Subagent（比如 A 查机票，B 查美食）。
在这个过程中，主 Agent 是**异步非阻塞**的，它不会干等着，而是充当“前台接待员”和“总线监控”，它可以继续跟用户闲聊安抚情绪，等所有 Subagent 汇报完结果后，主 Agent 再作为“最终整合器”把完美的结果输出给用户。
